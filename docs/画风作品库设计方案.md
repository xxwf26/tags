# 画风作品库 · 全新独立项目设计方案

> Codename: **style-atlas**（画风图谱）
> 版本：v1.0 ｜ 日期：2026-07-13

---

## 0. 一句话定位

**一个以「画风」为索引的作品发现引擎**：库里按多维画风标签检索作品→反查画师；库外按画风需求去小红书 / 微博 / 米画师主动爬作品和画师回来，AI 自动打标，人工复核入库或建联。

设计需求时想找"某类画风"，先在库里翻作品，并且按这个画风去外部平台爬新画师和作品回来，两项都要进行。

### 与传统画师库的区别

|  | 传统画师库 | 画风作品库（本项目） |
|---|---|---|
| 核心对象 | 画师 | **作品**（一个画师 N 个作品） |
| 画风标签打在 | 画师身上（粗粒度） | **每个作品**上（细粒度、多维） |
| 核心场景 | 按画师属性筛画师 | **按画风找作品 → 反查画师** |
| 数据来源 | 手动录入 | 手动 + **按画风主动去外部爬** |
| 打标 | 人工选 | **AI 自动打标**（人工复核） |

## 0.5 现有数据源：飞书台账分析与迁移

飞书多维表格 `2DLS-个人原画师`（248 条）是**现有唯一存量数据**，也是新系统的数据来源之一（后续主流量来自爬取）。其现状正是新系统要颠覆的对象，分析如下。

**表结构与字段映射**（飞书 → 新模型）：

| 飞书字段 | 类型 | 现状 | 映射到新模型 |
|---|---|---|---|
| `名称` | 文本 | 画师昵称 | `artists.name` |
| `个人概况` | 文本 | 小红书主页 URL | `artists.links.xiaohongshu` |
| `作品简历` | 文本 | 主页 URL 或单篇笔记链接 | `artists.links` / `artworks.source_url` |
| `风格备注` | 文本 | 自由文本，画风写在这里 | **废弃**，下沉为作品级 `artwork_tags`（见下） |
| `是否推进合作` | 单选 | 合作/不合作/待定/暂无档期/无法建联 | `artists.engage_status`（已对齐枚举） |
| `创建日期` | 日期 | 录入时间 | `artists.created_at` |
| `作品` | 附件 | 基本为空 | `artworks.image_url`（迁移时回爬补图） |

**`风格备注` 现状（新系统要解决的核心痛点）：**
- 画风只写在画师身上的自由文本，**不是作品级、无维度**；248 条中 **77% 为空**。
- 实际出现的画风词：国风(12)、日式(10)、国风·水墨(6)、二次元(5)、皮影(2)、水墨(1)、浅彩(1)、剪纸风(1)、年画(备注提及)——全是传统中国画风/民艺 + 日式 + 二次元。
- **同字段语义被污染**：档期（「25年档期已满」）、合作记录（「和XX合作了文创@柴门」）、主观评价（「风格较强，最好能全接」「人体不太稳定」）、✔ 标记、甚至附件 JSON 被误填。
- **命名不一致**：日式 vs 日系、国风 vs 国风古风；多值用「、」拼在一个单元格里（`国风、水墨`）。

**迁移方案**（一次性）：
1. 248 条画师按字段映射导入 `artists`（`风格备注` 里的档期/合作/评价分别落到 `engage_status` / `engage_note` / 丢弃，不进标签）。
2. 对每位画师回爬其小红书主页**一批作品**（覆盖画风分布），AI 打标后人工复核入库，把「画师身上一句画风」下沉为「作品级多维 `artwork_tags`」。
3. `风格备注` 里的画风词（国风/水墨/皮影/剪纸风/年画/日式）作为 P1 AI 打标的**别名归一**输入：日式→日系，国风/国风古风→国风，落进 §2.3 的 `genre` 子维度。

## 2. 数据模型

### 2.1 画师 `artists`

```sql
CREATE TABLE artists (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  name          VARCHAR(128) NOT NULL,          -- 画师名 / 昵称
  aliases       JSON,                           -- 别名（跨平台同一人）
  bio           TEXT,                           -- 简介
  contact       JSON,                           -- 联系方式 {wechat, qq, email...}
  links         JSON,                           -- 各平台主页 {xiaohongshu:[...], weibo:[...], mihuashi:[...]}
  commission    ENUM('open','full','commercial_only','unknown') DEFAULT 'unknown', -- 接稿状态
  drawing_habit JSON,                           -- 画画习惯（见 2.5）
  -- 合作/建联状态：对齐飞书台账「是否推进合作」真实枚举 + 保留建联推进阶段
  -- pending=待定 / cooperated=合作 / rejected=不合作 / no_availability=暂无档期 / unreachable=无法建联
  -- contacted=已接触 / negotiating=沟通中（建联推进中间态）
  engage_status ENUM('pending','contacted','negotiating','cooperated','rejected','no_availability','unreachable') DEFAULT 'pending',
  engage_note   TEXT,
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 2.2 作品 `artworks`（主角）

```sql
CREATE TABLE artworks (
  id             BIGINT PRIMARY KEY AUTO_INCREMENT,
  artist_id      BIGINT,                         -- FK artists（候选期可空）
  title          VARCHAR(255),
  image_url      VARCHAR(512) NOT NULL,          -- 本地 /uploads 原图
  thumb_url      VARCHAR(512),                   -- 缩略图（画廊性能）
  width          INT, height INT,
  orientation    ENUM('横','竖','方') DEFAULT '横', -- 朝向（width/height 派生；项目筛选：X6=横屏 / X3·恋语=竖屏）
  image_hash     CHAR(16),                       -- 感知哈希 pHash（跨平台去重）
  source_platform VARCHAR(32),                   -- xiaohongshu/weibo/mihuashi/manual
  source_url     VARCHAR(512),                   -- 原始链接（版权归属）
  ocr_text       TEXT,                           -- 图上文字（qwen-vl-ocr）
  ai_summary     TEXT,                           -- AI 画风描述
  ai_tagged      TINYINT DEFAULT 0,
  tag_confidence FLOAT,
  tag_status     ENUM('pending','confirmed') DEFAULT 'pending', -- 人工复核态
  embedding      JSON,                           -- 图像向量（以图搜图，P3）
  created_at     DATETIME DEFAULT CURRENT_TIMESTAMP,
  KEY idx_artist (artist_id),
  KEY idx_hash (image_hash),
  KEY idx_status (tag_status),
  KEY idx_orient (orientation)
);
```

### 2.3 标签体系（多维白名单，产品灵魂）

单一"擅长风格"标签不够用。拆成正交维度，每维都是**系统配置里的白名单**（配置=唯一数据源、删除即禁用），AI 只能从白名单里选，从源头杜绝脏标签。

```sql
-- 标签维度（两级：顶层维度 → 子维度；子维度 parent_id 指向顶层）
-- 顶层维度=正交大轴（genre/subject/...）；子维度=大轴内的分组（仅 genre 画风轴启用两级）
CREATE TABLE tag_dimensions (
  id        INT PRIMARY KEY AUTO_INCREMENT,
  parent_id INT,                          -- NULL=顶层维度；非 NULL=子维度（指向顶层 id）
  code      VARCHAR(32),                  -- 顶层 code 全局唯一(genre/subject/...)；子维度 code 父下唯一
  name      VARCHAR(64),                  -- 中文名
  sort      INT DEFAULT 0,
  KEY idx_parent (parent_id)
);

-- 标签值（白名单，AI 只能从这里选）
-- dimension_id 指向「最具体」的维度：有子维度时指子维度，否则指顶层
CREATE TABLE tags (
  id            INT PRIMARY KEY AUTO_INCREMENT,
  dimension_id  INT NOT NULL,             -- 所属维度（顶层或子维度）
  label         VARCHAR(64) NOT NULL,
  aliases       JSON,           -- 归一化别名（"古风" ← "国风古风","中国风"；"日式" ← "日系"）
  note          VARCHAR(255),
  enabled       TINYINT DEFAULT 1,   -- 删除 = 禁用（不物理删）
  UNIQUE KEY uq (dimension_id, label)
);

-- 作品 ↔ 标签 关联（多值走关联表，不用「、」拼字符串）
CREATE TABLE artwork_tags (
  artwork_id  BIGINT,
  tag_id      INT,
  source      ENUM('ai','manual') DEFAULT 'ai',  -- 谁打的
  confidence  FLOAT,
  PRIMARY KEY (artwork_id, tag_id)
);
```

> **按顶层维度筛选**：选「画风」类目时，`tags.dimension_id IN (顶层 genre 的 id ∪ 其子维度 id)`；其余顶层维度无子维度，直接按 `dimension_id` 匹配。

> **重要**：多值标签走关联表，不要用「、」拼接字符串——否则筛选时 `LIKE "%古风%"` 会误中"复古风"，是踩过的坑。

**维度结构：主画风轴两级 + 辅助小标签维度平级。** `genre`（画风）是主分类轴，下分**子维度**（画风分支），作品按此分类；`subject / technique / usage / tone / character` 是辅助小标签维度，平级、可随时新增。所有维度均为白名单，AI 只能从中选。

**初始维度种子：**

| 顶层 code | name | 子维度（仅 genre） | 初始标签示例 |
|---|---|---|---|
| `genre` | 画风 | 东方画风 | 国风、水墨、工笔、浮世绘 |
| | | 西方画风 | 欧美、油画、厚涂写实 |
| | | 二次元画风 | 日系、赛璐璐、二次元、像素 |
| | | 民艺 | 皮影、剪纸风、年画 |
| `subject` | 题材 | — | 人物立绘、场景、动物、Q版、机甲、美食、风景、建筑 |
| `technique` | 技法 | — | 线稿、平涂、厚涂、伪厚涂、噪点、描线 |
| `usage` | 用途 | — | 头像、立绘、表情包、漫画、插画、商稿、海报 |
| `tone` | 色调 / 情绪 | — | 高饱和、低饱和、冷色、暖色、治愈、性冷淡、通透 |
| `character` | 人物类型 | — | 成男、成女、小孩、Q版、非人 |

> 子维度与种子词表为初版，P1 AI 打标后从真实爬取数据反提补词、调分组。`genre` 的子维度把飞书台账里真实出现的 国风/水墨/皮影/剪纸风/年画 自然归位，并给「日式→日系」做别名归一。

这样"找某类画风"= 多维标签交叉筛选（如 `东方画风.国风 ∩ 立绘 ∩ 成男 ∩ 治愈`），比一个大杂烩标签精准得多。

> `character` 维度的来由：现有飞书台账的采编备注里反复出现「看不到男性角色」「没有成男作品」「只能画角色」这类**人物类型**诉求，单靠 `subject`（题材）无法表达，故独立成维。

## 3. 功能模块与页面

### 3.1 画风画廊 `GalleryPage`（核心入口）

- 顶部**多维标签筛选条**（横向，非左侧栏）：按维度分组排成一行行可折叠的标签 chip，交叉筛选（如 `东方画风.国风 ∩ 立绘 ∩ 成男 ∩ 治愈`）；同行多选为「或」、跨行为「与」。`genre` 维度按子维度（东方/西方/二次元/民艺）折叠展示
- **分级筛选**：一级 = 画风（genre，主轴）；二级 = 绘制技巧（technique）；其余（题材/用途/色调/人物类型）为辅助维度。筛选区按此顺序排列，避免标签一多糊成一团
- **朝向归类**（业务点1）：筛选条含「全部 / 横屏 / 竖屏」切换；选「全部」时作品按横屏区、竖屏区**分组归类展示**，方便按项目取用（X6=横屏 / X3·恋语=竖屏）。`artworks.orientation` 落库加索引
- **标签前期可用**（业务点2）：多维标签不藏在详情弹层里，而是作为筛选 chip 直接出现在画风分布下方，前期筛选即可用
- 默认浏览模式：**作品按画风（genre 子维度）分类**——主区可切换「瀑布流 / 按画风分组」两种视图，分组视图下每个画风分支一个区块，点进即筛
- 主区**作品瀑布流**（虚拟滚动，缩略图），占满筛选条下方整宽
- 筛选条内集成关键词 + 排序（最新 / 最相似）
- **大图查看**（业务点4）：① 缩略图鼠标悬浮自动放大预览；② 缩略图角部「⤢」按钮点开**全屏查看器**，显示大图 + 全维标签；③ 全屏内 ←/→ 键翻上一张/下一张，**限定在当前画风+标签+朝向筛选范围内**，不跳到别的画风；ESC 关闭
- 点作品 → **Lightbox 大图** + 全维标签 + 「查看画师」跳转
- 进阶：**以图搜图**——上传 / 选中一张图，找库内相似画风（P3，图像向量）

### 3.2 画师详情 `ArtistPage`

- 画师画像 + 联系方式 + 建联状态 + **画画习惯卡片**
- **作品按画风分组**：按该画师作品上的 `genre` 标签聚类成组（如「国风 12 张」「二次元 5 张」），每组展示代表作缩略图——直接坐实「这个画师的每种画风都有对应作品」，缺作品的画风组标「待补」
- 该画师的画风标签聚合（从其作品标签统计出高频画风，作为分组依据）
- 顶部**画风分布条**：各画风作品数占比，点条筛选到对应分组
- 画风分布下方**多维标签筛选区**（分级：一级画风 / 二级绘制技巧 / 辅助维度），多选叠加筛选该画师作品
- **朝向归类**：作品区按横屏/竖屏分组展示 + 朝向切换（同 §3.1）
- **大图查看器**：悬浮放大 + 全屏 + 筛选范围内 ←/→ 翻图（同 §3.1）

### 3.3 采集 / 发现 `DiscoverPage`

- 输入画风需求 → 一键生成各平台搜索链接 / 触发抓取
- **候选复核队列**：卡片展示抓到的画师 + 作品 + AI 标签候选，按钮「转正入库 / 合并到已有画师 / 建联 / 丢弃」
- 抓取历史 / 进度面板（running 时轮询）

### 3.4 标签配置 `ConfigPage`

- 管理 6 个维度及其标签白名单（增 / 禁用 / 别名）
- **删除 = 禁用**（有作品在用则拒删，引用保护）
- 新标签可从复核时"提升"进白名单

## 4. 采集 + AI 打标管线（后端核心）

```
【数据来源】
入库数据是多源汇聚，飞书台账「2DLS-个人原画师」只是其中一种（历史人工录入，画师级自由文本画风）；
主流量来自按画风主动爬取（小红书 / 微博 / 米画师）。飞书台账需做一次性迁移：画师入库 + 其小红书作品回爬打标，
把「画师身上一句画风」下沉为「作品级多维标签」。

【按画风采集】
DiscoverPage 输入"日系治愈立绘"
  → crawl/pipeline 生成平台搜索 + 抓取
      · 小红书：SSR 法（fetchHtml + 抠 __INITIAL_STATE__，无登录、不触登录墙）
      · 米画师：playwright 开主页拦免签 JSON
      · 微博：需 PoC（见风险 5）
  → 每个画师拉「一批」作品（不只一张），覆盖其画风分布，保证每种画风都有代表作坐实
  → 每个作品图 → tagging 模块（**Gemini + 豆包 双模型集成打标**，见下）
      · gemini-3.5-flash（Gemini 原生 v1beta，responseMimeType=json）+ doubao-seed-1-6-vision（OpenAI 兼容）各打一遍
      · normalize：两路 AI 标签 → 映射白名单（命中才留，别名归一）
      · 集成：两模型都选 = 高置信自动入库；仅一方选 = 低置信进人工复核；都不选 = 不打该标签
      · 生成 ai_summary + tag_confidence（共识置信度高于单模型）
  → extractor(LlmClient) 聚合画师画像 + drawing_habit + 画风分布（按作品 genre 标签统计）
  → 落 candidates(status=pending)，作品图暂存 CDN 原链（不落盘）
  → 人工在 DiscoverPage 复核 → promote：画师 + 作品入库、图片下载落 /uploads

【存量批量打标】
选中未打标作品 → tagging 批量（Gemini+豆包集成）→ tag_status=pending → 人工确认
```

**AI 打标一致性关键**：提示词里把 §2.3 的维度和候选值全塞进去，强约束模型「只从给定词表选、输出 JSON、命中不了就留空」，从源头对齐白名单、不产生脏标签。支持两种触发：①存量作品**批量打标**；②抓取入库时**顺带打标**。

**模型选型（Gemini + 豆包集成）**：verify/ 横评结论（13 张真实小红书图，共识伪金标准）——doubao-seed-1-6-vision F1 87.1% 最稳、题材 100%/色调 92.1%；gemini-3.5-flash F1 78.7%、13/13 成功、~3s/图。两者互补：豆包准而稳、Gemini 快且对水彩/扁平等发散判断有补充。故打标用**双模型集成**：共识标签高置信自动入库、分歧标签人工复核。glm-5v-turbo 解析率低（3/13）已弃用；gpt-5.5 题材强（F1 91.9%）但风格/技法发散、且成本高，暂不纳入主链路。

**共享 AI 客户端：**

- `LlmClient`：Anthropic 协议走中转，剥 thinking、重试、分块
- `DoubaoVisionClient`：OpenAI 兼容 `/v1/chat/completions` 调 `doubao-seed-1-6-vision`（中转 tc-paperhub）
- `GeminiVisionClient`：Gemini 原生 `/gemini/v1beta/models/<model>:generateContent` + `inline_data` base64 + `responseMimeType:application/json` 调 `gemini-3.5-flash`
- `OcrClient`：OpenAI 兼容调 `qwen-vl-ocr`（仅读图上文字，不做画风判断）
- 打标 = `GeminiVisionClient` + `DoubaoVisionClient` 集成（见上）

## 7. 风险与要拍板的点

1. **⚠️ 版权与存储**：作品库会大量存他人作品图。建议只存缩略图 + 必留 `source_url` 和画师署名，定性为"仅内部找参考 / 建联、不商用"。量级大时 `/uploads` 存储成本要评估。
2. **微博抓取未验证**：小红书 / 米画师已跑通，微博有风控 / 登录墙，需单独 PoC，别默认好抓。P2 先只上小红书。
3. **以图搜图 vs MySQL**：MySQL 8 无原生向量类型。方案：先把 embedding 存 JSON、小规模应用层算余弦；量大再引专门向量库。放 P3。
4. **标签维度按 6 维起步**（genre / subject / technique / usage / tone / character）：`character`（人物类型）因采编实际诉求新增；表结构已支持任意加维，初始词表沿用种子，P1 AI 打标后再从真实数据反提补词。
5. **中转 AI 的 key / 额度**：批量打标会吃视觉模型 token，量大时留意额度。
