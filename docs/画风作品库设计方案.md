# 画风作品库 · 全新独立项目设计方案

> Codename: **style-atlas**（画风图谱）
> 版本：v1.0 ｜ 日期：2026-07-13

---

## 0. 一句话定位

**一个以「画风」为索引的作品发现引擎**：库里按多维画风标签检索作品→反查画师；库里不够，就按画风需求去小红书 / 微博 / 米画师主动爬作品和画师回来，AI 自动打标，人工复核入库或建联。

设计需求时想找"某类画风"，先在库里翻作品，翻不到就按这个画风去外部平台爬新画师和作品回来。

### 与传统画师库的区别

|  | 传统画师库 | 画风作品库（本项目） |
|---|---|---|
| 核心对象 | 画师 | **作品**（一个画师 N 个作品） |
| 画风标签打在 | 画师身上（粗粒度） | **每个作品**上（细粒度、多维） |
| 核心场景 | 按画师属性筛画师 | **按画风找作品 → 反查画师** |
| 数据来源 | 手动录入 | 手动 + **按画风主动去外部爬** |
| 打标 | 人工选 | **AI 自动打标**（人工复核） |

## 1. 技术栈与工程结构

独立仓库、独立库、独立端口。

- **前端**：React 19 + TS + Vite + Tailwind4 + shadcn/ui + react-query

## 2. 数据模型

### 2.1 画师 `artists`

```sql
CREATE TABLE artists (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  name          VARCHAR(128) NOT NULL,          -- 画师名 / 昵称
  aliases       JSON,                           -- 别名（跨平台同一人）
  bio           TEXT,                           -- 简介
  contact       JSON,                           -- 联系方式 {wechat, qq, email...}
  links         JSON,                           -- 各平台主页 {xiaohongshu:[...], weibo:[...], mihuashi:[...]}
  commission    ENUM('open','full','commercial_only','unknown') DEFAULT 'unknown', -- 接稿状态
  drawing_habit JSON,                           -- 画画习惯（见 2.5）
  engage_status ENUM('none','contacted','negotiating','cooperated','rejected') DEFAULT 'none', -- 建联状态
  engage_note   TEXT,
  source        VARCHAR(32),                    -- 首次来源平台
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 2.2 作品 `artworks`（主角）

```sql
CREATE TABLE artworks (
  id             BIGINT PRIMARY KEY AUTO_INCREMENT,
  artist_id      BIGINT,                         -- FK artists（候选期可空）
  title          VARCHAR(255),
  image_url      VARCHAR(512) NOT NULL,          -- 本地 /uploads 原图
  thumb_url      VARCHAR(512),                   -- 缩略图（画廊性能）
  width          INT, height INT,
  image_hash     CHAR(16),                       -- 感知哈希 pHash（跨平台去重）
  source_platform VARCHAR(32),                   -- xiaohongshu/weibo/mihuashi/manual
  source_url     VARCHAR(512),                   -- 原始链接（版权归属）
  ocr_text       TEXT,                           -- 图上文字（qwen-vl-ocr）
  ai_summary     TEXT,                           -- AI 画风描述
  ai_tagged      TINYINT DEFAULT 0,
  tag_confidence FLOAT,
  tag_status     ENUM('pending','confirmed') DEFAULT 'pending', -- 人工复核态
  embedding      JSON,                           -- 图像向量（以图搜图，P3）
  created_at     DATETIME DEFAULT CURRENT_TIMESTAMP,
  KEY idx_artist (artist_id),
  KEY idx_hash (image_hash),
  KEY idx_status (tag_status)
);
```

### 2.3 标签体系（多维白名单，产品灵魂）

单一"擅长风格"标签不够用。拆成正交维度，每维都是**系统配置里的白名单**（配置=唯一数据源、删除即禁用），AI 只能从白名单里选，从源头杜绝脏标签。

```sql
-- 标签维度：风格流派 / 题材 / 技法 / 用途 / 色调 ...
CREATE TABLE tag_dimensions (
  id       INT PRIMARY KEY AUTO_INCREMENT,
  code     VARCHAR(32) UNIQUE,   -- genre / subject / technique / usage / tone
  name     VARCHAR(64),          -- 中文名
  sort     INT DEFAULT 0
);

-- 标签值（白名单，AI 只能从这里选）
CREATE TABLE tags (
  id            INT PRIMARY KEY AUTO_INCREMENT,
  dimension_id  INT NOT NULL,
  label         VARCHAR(64) NOT NULL,
  aliases       JSON,           -- 归一化别名（"古风" ← "国风古风","中国风"）
  note          VARCHAR(255),
  enabled       TINYINT DEFAULT 1,   -- 删除 = 禁用（不物理删）
  UNIQUE KEY uq (dimension_id, label)
);

-- 作品 ↔ 标签 关联（多值走关联表，不用「、」拼字符串）
CREATE TABLE artwork_tags (
  artwork_id  BIGINT,
  tag_id      INT,
  source      ENUM('ai','manual') DEFAULT 'ai',  -- 谁打的
  confidence  FLOAT,
  PRIMARY KEY (artwork_id, tag_id)
);
```

> **重要**：多值标签走关联表，不要用「、」拼接字符串——否则筛选时 `LIKE "%古风%"` 会误中"复古风"，是踩过的坑。

**初始维度种子：**

| dimension.code | name | 初始标签示例 |
|---|---|---|
| `genre` | 风格流派 | 日系、厚涂、赛璐璐、国风古风、欧美、像素、扁平、水彩、暗黑、二次元 |
| `subject` | 题材 | 人物立绘、场景、动物、Q版、机甲、美食、风景、建筑 |
| `technique` | 技法 | 线稿、平涂、厚涂、伪厚涂、噪点、描线 |
| `usage` | 用途 | 头像、立绘、表情包、漫画、插画、商稿、海报 |
| `tone` | 色调 / 情绪 | 高饱和、低饱和、冷色、暖色、治愈、性冷淡、通透 |

这样"找某类画风"= 多维标签交叉筛选（如 `日系 ∩ 立绘 ∩ 治愈`），比一个大杂烩标签精准得多。

### 2.4 采集相关

```sql
-- 采集任务（一次"按画风爬"）
CREATE TABLE crawl_runs (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  query       VARCHAR(255),      -- 画风需求，如"日系治愈立绘"
  platform    VARCHAR(32),
  status      ENUM('running','ok','failed') DEFAULT 'running',
  found_count INT DEFAULT 0,
  message     TEXT,
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 候选（画师 + 作品一起进复核队列）
CREATE TABLE candidates (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  run_id        BIGINT,
  artist_name   VARCHAR(128),
  source_url    VARCHAR(512),
  raw           JSON,           -- 抓到的原始画像 + 作品图 + AI 标签候选
  dedup_artist_id BIGINT,       -- 疑似已存在画师
  status        ENUM('pending','promoted','merged','rejected') DEFAULT 'pending',
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 2.5 `drawing_habit` JSON 结构（画画习惯）

"画画习惯"= 帮助匹配和建联的行为画像，由同一画师**多篇作品 / 笔记聚合**推断，周期性重爬刷新。

```json
{
  "update_frequency": "高产/月更/随缘",
  "frequent_subjects": ["原创女性角色", "古风"],
  "active_platforms": ["xiaohongshu", "weibo"],
  "active_time": "晚间更新居多",
  "style_trend": "早期偏赛璐璐，近期转厚涂",
  "commission_signal": "主页写'约稿开放'",
  "last_analyzed_at": "2026-07-13"
}
```

## 3. 功能模块与页面

### 3.1 画风画廊 `GalleryPage`（核心入口）

- 左侧**多维标签筛选面板**：按 5 个维度分组勾选，交叉筛选（如 `日系 ∩ 立绘 ∩ 治愈`）
- 主区**作品瀑布流**（虚拟滚动，缩略图）
- 顶部关键词 + 排序（最新 / 最相似）
- 点作品 → **Lightbox 大图** + 全维标签 + 「查看画师」跳转
- 进阶：**以图搜图**——上传 / 选中一张图，找库内相似画风（P3，图像向量）

### 3.2 画师详情 `ArtistPage`

- 画师画像 + 联系方式 + 建联状态 + **画画习惯卡片**
- 该画师全部作品九宫格
- 该画师的画风标签聚合（从其作品标签统计出高频画风）

### 3.3 采集 / 发现 `DiscoverPage`

- 输入画风需求 → 一键生成各平台搜索链接 / 触发抓取
- **候选复核队列**：卡片展示抓到的画师 + 作品 + AI 标签候选，按钮「转正入库 / 合并到已有画师 / 建联 / 丢弃」
- 抓取历史 / 进度面板（running 时轮询）

### 3.4 标签配置 `ConfigPage`

- 管理 5 个维度及其标签白名单（增 / 禁用 / 别名）
- **删除 = 禁用**（有作品在用则拒删，引用保护）
- 新标签可从复核时"提升"进白名单

## 4. 采集 + AI 打标管线（后端核心）

```
【按画风采集】
DiscoverPage 输入"日系治愈立绘"
  → crawl/pipeline 生成平台搜索 + 抓取
      · 小红书：SSR 法（fetchHtml + 抠 __INITIAL_STATE__，无登录、不触登录墙）
      · 米画师：playwright 开主页拦免签 JSON
      · 微博：需 PoC（见风险 5）
  → 每个作品图 → tagging 模块
      · VisionClient(qwen-vl-ocr / 视觉模型) 读图 → 按【固定标签体系】输出多维候选
      · normalize：AI 标签 → 映射白名单（命中才留，别名归一）
      · 生成 ai_summary + tag_confidence
  → extractor(LlmClient) 聚合画师画像 + drawing_habit
  → 落 candidates(status=pending)，作品图暂存 CDN 原链（不落盘）
  → 人工在 DiscoverPage 复核 → promote：画师 + 作品入库、图片下载落 /uploads

【存量批量打标】
选中未打标作品 → tagging 批量 → tag_status=pending → 人工确认
```

**AI 打标一致性关键**：提示词里把 §2.3 的维度和候选值全塞进去，强约束模型「只从给定词表选、输出 JSON、命中不了就留空」，从源头对齐白名单、不产生脏标签。支持两种触发：①存量作品**批量打标**；②抓取入库时**顺带打标**。

**共享 AI 客户端：**

- `LlmClient`：Anthropic 协议走中转，剥 thinking、重试、分块
- `VisionClient`：OpenAI 兼容 `/v1/chat/completions` 调 `qwen-vl-ocr`（读图上文字 + 画风判断）

```
POST /api/crawl/run           # {query, platform}
GET  /api/crawl/runs
GET  /api/candidates?status=pending
POST /api/candidates/:id/promote|merge|reject
```

## 6. 分期路线图

| 阶段 | 交付 | 说明 |
|---|---|---|
| **P0 地基** | 工程脚手架 + 3 张核心表(artists/artworks/tags) + 标签维度种子 + 手动录作品 + **画风画廊多维筛选** | 先让"库里翻画风"跑起来 |
| **P1 AI 打标** | VisionClient + 打标管线 + 白名单归一 + 复核态 + 存量批量打标 | 解决"没人工打标" |
| **P2 外部采集** | crawl 管线（小红书先行）+ candidates 复核队列 + 转正入库 + 画画习惯聚合 | 按画风向外扩库 |
| **P3 增强** | 以图搜图（图像向量）+ 微博接入 + 周期重爬刷新习惯 + pHash 去重 | 终极形态 |

建议 P0 先做，能最快看到"以画风找作品"的核心价值。

## 7. 风险与要拍板的点

1. **⚠️ 版权与存储**：作品库会大量存他人作品图。建议只存缩略图 + 必留 `source_url` 和画师署名，定性为"仅内部找参考 / 建联、不商用"。量级大时 `/uploads` 存储成本要评估。
2. **微博抓取未验证**：小红书 / 米画师已跑通，微博有风控 / 登录墙，需单独 PoC，别默认好抓。P2 先只上小红书。
3. **以图搜图 vs MySQL**：MySQL 8 无原生向量类型。方案：先把 embedding 存 JSON、小规模应用层算余弦；量大再引专门向量库。放 P3。
4. **标签维度是否就按 5 维起步**：这是产品核心内容，需确认维度和初始词表；也可先只做 `genre` 一维快速验证，后续扩维（表结构已支持任意加维）。
5. **中转 AI 的 key / 额度**：批量打标会吃视觉模型 token，量大时留意额度。
