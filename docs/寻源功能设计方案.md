# 寻源功能 · 产品设计方案（v2 重构提案）

> 面向需求：**上传一张图 或 设置画风标签 → 在小红书/微博/米画师搜到相关帖子 → 保证结果质量**。
> 本文是对现有「寻源」模块（关键词搜+AI过滤，效果不达预期）的重新设计，供拍板后实施。
> 关联主设计：[画风作品库设计方案.md](画风作品库设计方案.md)（§2.3 标签体系、§4 采集打标管线、§7 风险）。

> **【v2.1 更新 · 已实测拍板】** 需求锁定为「**上传图 → 找画风相近的帖子，且单次召回量大**」。经实测确定技术路径（详见文末 §11 实测附录），要点：
> - ❌ AI 网关 `tc-paperhub` **无图像 embedding 模型**（仅文本 embedding），不能用它算画风向量。
> - ✅ **本地 CLIP**（`@xenova/transformers` + `Xenova/clip-vit-base-patch32`）已在本机跑通：512 维、单图 ~300ms、能区分画风远近（相似度 0.58~0.78 有梯度）。**画风向量走本地 CLIP**。
> - ⚠️ **发现真凶**：此前 `server/.env` 无 `AI_API_KEY`，导致所有 AI 调用 401、`gateArtwork` 一路中性放行（quality=5 全过）——这是"搜出来质量差"的头号原因。key 已补，AI 质检现已生效。
> - 架构：单次召回量大 → **必须异步任务队列**（同步必超时）。

## 1. 先厘清能做到什么、做不到什么（诚实边界）

三个目标平台**都不提供反向图搜 / 向量检索 API**。因此：

| 诉求 | 可行性 | 说明 |
|---|---|---|
| 上传图 → 找到**这张图的出处/同一张图** | ❌ 不可行 | 平台无反向图搜接口，无法以图反查 |
| 上传图 → 找到**画风/题材相近**的帖子 | ✅ 可行 | 图→AI标签→关键词搜→**用参考图视觉精排** |
| 设置标签 → 搜相关帖子 | ✅ 可行 | 直接关键词搜 |
| **保证结果质量** | ✅ 可控 | 靠我们自己的**AI质检闸门 + 视觉相似度精排**，非平台能力 |

**结论**：把"寻源"定位为 **「以画风/参考图为线索，跨平台发现候选作品，并用两道自有关卡保证质量」**，而不是"精确图搜"。这是三平台约束下能达到的最佳形态。

## 2. 核心思路：三段式管线（理解 → 检索 → 精排质检）

```
                    ┌── 入口A：上传参考图 ──┐
                    │                        ├─→【理解】→【检索】→【精排+质检】→ 候选队列 → 复核入库
                    └── 入口B：选画风标签 ──┘
```

**两个入口统一汇入同一条管线**，区别只在「理解」阶段：

### 阶段1 · 理解（Understand）—— 产出「搜索标签 + 可选参考向量」
- **入口A（上传图）**：图 → Gemini+豆包双模型打标（复用 §4 现有打标链路）→ 命中白名单的画风/题材标签，**预勾选**给用户；同时对参考图算视觉特征向量（见 §5 精排）。用户可增删标签后再搜。
- **入口B（选标签）**：用户直接从词表勾画风/题材标签。无参考向量。
- 两者最终都得到一组**搜索关键词**（画风词为主，题材词辅助）。

### 阶段2 · 检索（Retrieve）—— 平台关键词搜，广撒网
- 对每个选中平台 × 每个关键词，调平台搜索抓一批帖子（小红书/微博/米画师）。
- 这一步**只求召回、不求精度**，故意多抓（如每词 30~50 条）作为候选池。
- 平台可多选；每平台可用性见 §7。

### 阶段3 · 精排 + 质检（Re-rank & Gate）—— 两道关卡保证质量
候选池逐张过两道关，这是"保证质量"的核心：
1. **AI 质检闸门**（复用 `gateArtwork`）：判定 `is_artwork`（剔除广告/照片/文字海报/表情包）+ 打 `quality` 质量分（0-10），低于阈值直接淘汰。
2. **视觉相似度精排**（仅入口A有参考图时）：对通过闸门的图算视觉向量，与参考图算余弦相似度，**按相似度降序**排列；可设相似度下限，太不像的淘汰。
3. **pHash 去重**：与库内已有作品 + 本次结果去近重复。

最终产出：**已过滤、已按"像不像参考图 + 质量分"排序**的候选列表，入候选队列由人工复核转正。

## 3. 质量保证机制（分层，逐层收紧）

| 层 | 机制 | 淘汰什么 | 依赖 |
|---|---|---|---|
| L1 关键词 | 画风+题材词组合搜 | 主题无关 | 现有 |
| L2 AI闸门 | `gateArtwork` 判绘画+质量分 | 广告/照片/海报/低质 | 现有（单模型，省成本）|
| L3 视觉精排 | 参考图余弦相似度排序+下限 | 与参考图风格不符 | **需 embedding（见 §5）**|
| L4 去重 | pHash 海明距离 | 重复/已入库 | 现有 imghash |
| L5 人工 | 复核队列 promote/reject | 前几层漏网 | 现有候选队列 |

> 「保证质量」= L2+L3+L5 三重收口，而非追求 100% 精确。诚实预期：**结果是"高质量的相关候选线索"，最终由人拍板**。

## 4. 数据流与页面交互

**寻源页（SearchPage / 或并入采集页）**：
```
┌─────────────────────────────────────────────┐
│ [拖/粘贴/选 参考图]  或  [不传图]              │  ← 入口A / B
├─────────────────────────────────────────────┤
│ 画风标签：⬤日系 ⬤治愈  ○厚涂 ...（AI预勾/手选）│
│ 平台：☑米画师 ☑小红书 ☑微博                    │
│ 每词数量：[30]   相似度下限：[——●——] 60%      │  ← 下限仅传图时有效
│                              [🔍 开始寻源]     │
├─────────────────────────────────────────────┤
│ 结果（按 相似度×质量 排序，N 张）              │
│ ┌────┐ ┌────┐ ┌────┐   每张卡：                │
│ │图  │ │图  │ │图  │   · 画师名 + 来源平台      │
│ │92% │ │88% │ │85% │   · 相似度% + 质量分       │  ← 点击看大图/原帖
│ └────┘ └────┘ └────┘   · [复核] [入库] [丢弃]  │
└─────────────────────────────────────────────┘
```
- 结果卡显示**画师、来源、相似度、质量分**，缩略图**可点开大图**（多图翻页 + 跳原帖）。
- 复核转正复用现有候选队列 → 下载图、建/选画师、入库、AI打标。

## 5. 关键技术依赖：视觉相似度精排（L3）

这是"上传图保证质量"的灵魂，也是**唯一需要新增能力**的部分。三种实现档位，按投入递增：

| 档位 | 方案 | 效果 | 成本/依赖 |
|---|---|---|---|
| **A（最小）** | 不做视觉精排，仅 L2 质检 + 平台热度/点赞排序 | 保证"是好画"，但不保证"像参考图" | 零新依赖，可立即做 |
| **B（推荐）** | 接入**多模态 embedding**（CLIP 或 AI 网关的 embedding 接口）对参考图+候选图算余弦相似度 | 真正"找相近画风"，质量与相关性双保证 | **需确认 AI 网关是否有 embedding 接口**，或本地跑 CLIP |
| C（重） | 自建向量库（量大后）| 规模化 | Milvus/pgvector 等，P3+ |

> 现有 `imghash`（aHash/pHash）**只能测近重复，不能测风格相似**，无法胜任 L3。
> **待确认（拍板点1）**：`tc-paperhub` 网关是否提供 embedding 接口？若有→档位B可直接做；若无→先上档位A，或引本地 CLIP。

MySQL 无原生向量类型：参考图向量存 `reference_images.embedding`（JSON），候选图向量算完即用不落库（量小，应用层余弦）。与主设计 §7.3 一致。

## 9. 风险与取舍

1. **精确图搜做不到**——已在 §1 说明，产品话术需诚实（"发现相近作品"而非"找同款"）。
2. **embedding 依赖未定**——决定 S2 能否做、走网关还是本地 CLIP（拍板点1）。
3. **同步阻塞**——一次寻源要搜多平台+下载+AI，耗时长。S1 可先同步（提示耗时），量大后改任务队列+进度（拍板点2）。
4. **版权/存储**——沿用主设计 §7.1：候选阶段只存原链不落盘，入库才下载。
5. **小红书 cookie 维护**——会过期，需人工更新，或接受小红书时好时坏。

## 11. 实测附录与最终落地方案（v2.1，已验证）

### 11.1 实测结论（本机真实运行）
| 项 | 结果 |
|---|---|
| 网关图像 embedding | ❌ 无（63 模型中 embedding 全为文本 `text-embedding-*`；`multimodal-embedding-*` 等 4 名皆 404）|
| 网关文本 embedding | ✅ `text-embedding-v4` 1024维可用（本方案不采用）|
| 本地 CLIP | ✅ `Xenova/clip-vit-base-patch32`：512维、加载~14s、单图~300ms、画风相似度有梯度(0.58~0.78) |
| AI 质检 gemini | ✅ 补 key 后 `is_artwork/quality` 正常（此前 401 静默失效=质差真凶）|

### 11.2 最终架构：本地 CLIP 精排 + 异步任务
```
寻源请求（图 or 标签 + 平台 + 每词数量 + 相似度下限）
  → 建 search_task（status=running）→ 立即返回 taskId
  → 【后台异步】：
     1. 理解：入口A图→CLIP参考向量 + AI打标预选标签；入口B直接用标签词
     2. 检索：三平台 × 关键词 广召回候选池（去重 sourceUrl）
     3. 逐张精排质检（分批并发，写进度 done/total）：
        下载 → gateArtwork(is_artwork+quality) 过闸 → CLIP向量 → 与参考图余弦相似度
     4. 排序：入口A按 相似度×质量 降序；入口B按 质量 降序
     5. 写结果 + status=done
  → 前端轮询 GET /api/search/task/:id 进度条 → 完成出结果卡
  → 复核转正走候选队列（复用现有 promote）
```

### 11.3 组件与复用
- **新增** `server/src/modules/embed/clip.ts`：单例懒加载 CLIP，`embedImage(buf)→number[512]`（L2归一化）；`cosine(a,b)`。
- **复用**：`gateArtwork`（质检）、`imghash`（去重）、`candidates`（复核队列）、三平台 search（`searchMihuashi`/`searchWeiboByKeyword`/`searchXhsByKeyword`）、`crawl/weibo` CORS 修法、米画师详情页抓作者。
- **任务表**：复用 `search_sessions`（加 `done_count`/`total_count`/`ref_embedding` 或存 raw JSON），结果进 `candidates.raw`（带 similarity/quality），**弃用 search_results 三级库**（拍板点3）。
- 参考图向量存 `reference_images.embedding`（JSON）。

### 11.4 「量大」的性能账（单次召回 N 张）
- 耗时 ≈ N ×（下载 0.5s + gate 1~2s + CLIP 0.3s）。**AI 质检是主瓶颈**（每张一次 gemini）。
- 1000 张 ≈ 25~40 分钟 → **必须异步 + 分批并发（如 5~8 并发）+ 进度条**。
- 成本：CLIP 本地零成本；gate 每张一次 gemini 调用，1000张=1000次，按额度评估（可选：先 CLIP 粗排取 Top-K 再对 Top-K 过 gate，省 AI 调用）。
  - **优化建议**：图多时先 CLIP 相似度粗排 → 只对 Top-K（如前 100）跑 AI 质检，AI 调用量从 N 降到 K，大幅省额度、提速。
